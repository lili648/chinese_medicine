### 2.2 Level 1 Design Description 第1层设计描述

#### 2.2.1 System Architecture 系统结构

##### 2.2.1.1 Description of the Architecture 系统结构描述

系统整体采用 **三层架构 + 管道模式**。数据从原始文献文件流入，经预处理 → NER → 图谱构建管道逐步加工，最终通过 Web 层对外服务。

```
                      ┌─────────────┐
                      │   浏览器      │
                      │ (Chrome/Edge)│
                      └──────┬──────┘
                             │ HTTP
                      ┌──────▼──────┐
                      │  Streamlit   │
                      │  (Port 8501) │
                      └──────┬──────┘
                             │
                  ┌──────────┴──────────┐
                  │                     │
           ┌──────▼──────┐      ┌──────▼──────┐
           │ Flask API    │      │ Streamlit   │
           │ (Port 5000)  │      │ Pages       │
           │  /api/query  │      │ 直接调用     │
           │  /api/search │      │             │
           └──────┬──────┘      └──────┬──────┘
                  │                    │
          ┌───────┴────────────────────┴───────┐
          │          业务逻辑层                  │
          │  ┌──────────┐  ┌──────────────────┐│
          │  │Preprocessor│  │ GraphBuilder     ││
          │  │NER Engine  │  │ QueryService     ││
          │  └──────────┘  └──────────────────┘│
          └───────┬────────────────┬───────────┘
                  │                │
          ┌───────▼──────┐  ┌──────▼──────┐
          │  文件系统      │  │  MySQL DB   │
          │  data/ dicts/ │  │  (TCP 3306) │
          │  output/      │  │             │
          └──────────────┘  └─────────────┘
```

**组件间通信方式：**

| 通信路径 | 方式 | 说明 |
|----------|------|------|
| 浏览器 ↔ Streamlit | HTTP/WebSocket | Streamlit 内置通信 |
| Streamlit Pages ↔ 业务层 | Python 直接调用 | 同进程内函数调用 |
| Streamlit ↔ Flask API | HTTP REST (JSON) | 跨进程调用，用于图谱查询 |
| 业务层 ↔ MySQL | TCP/IP 协议 (pymysql) | 关系型数据库访问 |
| NER 输出 → 图谱输入 | JSON 文件 | 管道中间产物，解耦两个模块 |

**图计算设计：** MySQL 负责持久化存储实体与关系数据，NetworkX 负责图算法计算（最短路径、邻居查询等）。系统启动时从 MySQL 加载数据到 NetworkX 构建内存图，查询全部走 NetworkX API。MySQL 不可用时，系统降级为纯 NetworkX 内存模式（从 JSON 文件加载数据）。Web 层通过统一的 QueryService 接口调用，不受底层存储切换影响。

##### 2.2.2.1 模块一：文献数据导入与预处理模块

###### 2.2.2.1.1 文献数据加载（F-01）

**1 功能设计描述**

负责读取原始文献数据文件，解析为标准化的 Article 对象列表。支持 CSV 和 JSON 两种格式。对格式异常或缺失关键字段的文献进行日志记录和跳过处理，不中断整体加载流程。

**（1）类**

**1）Article（数据类）**

| 属性 | 类型 | 说明 |
|------|------|------|
| article_id | str | 文献唯一标识 |
| pmid | str | PubMed ID（英文文献） |
| title | str | 标题 |
| abstract | str | 摘要 |
| authors | str | 作者列表 |
| journal | str | 期刊名称 |
| pub_year | int | 出版年份 |
| language | str | 语种（en/zh） |
| source_file | str | 来源文件名 |
| tokens | list[str] | 分词结果（预处理后填充） |

```python
@dataclass
class Article:
    article_id: str
    pmid: Optional[str]
    title: str
    abstract: Optional[str]
    authors: Optional[str]
    journal: Optional[str]
    pub_year: Optional[int]
    language: str        # "en" or "zh"
    source_file: str
    tokens: List[str] = field(default_factory=list)
```

**2）DataLoader（数据加载器）**

| 方法 | 说明 |
|------|------|
| `load_csv(file_path) -> List[Article]` | 从 CSV 加载文献 |
| `load_json(file_path) -> List[Article]` | 从 JSON 加载文献 |
| `load_directory(dir_path) -> List[Article]` | 批量加载目录下所有文献文件 |
| `_parse_row(row) -> Article` | 单行数据解析 |
| `_handle_missing(article) -> Article` | 缺失值处理，标注空字段 |

**（2）文件列表**

| 名称 | 类型 | 存放位置 | 说明 |
|------|------|----------|------|
| data_loader.py | Python | src/preprocessing/ | 文献数据加载器，含 Article 数据类和 DataLoader 类 |
| __init__.py | Python | src/preprocessing/ | 模块初始化文件 |

###### 2.2.2.1.2 文本预处理流水线（F-02）

**1 功能设计描述**

对文献的 title 和 abstract 字段执行文本清洗流水线：去除 HTML 标签 → 全角转半角 → 去特殊字符和数字 → 去停用词 → jieba 分词。英文文献额外进行大小写归一化。最终将分词结果存入 Article.tokens。

**（1）类**

**1）Preprocessor（文本预处理器）**

| 方法 | 说明 |
|------|------|
| `clean_text(text: str) -> str` | 文本清洗：去HTML/特殊字符/数字/停用词 |
| `segment(text: str, lang: str) -> List[str]` | 分词：中文用jieba，英文按空格+大小写归一化 |
| `process_article(article: Article) -> Article` | 处理单篇文献，返回填充了 tokens 的对象 |
| `process_batch(articles: List[Article]) -> List[Article]` | 批量处理文献列表 |

**2）StopWords（停用词管理器）**

| 方法 | 说明 |
|------|------|
| `load(file_path: str) -> None` | 加载停用词表 |
| `is_stopword(word: str) -> bool` | 判断是否为停用词 |
| `remove(text_list: List[str]) -> List[str]` | 从词列表中移除停用词 |

**（2）文件列表**

| 名称 | 类型 | 存放位置 | 说明 |
|------|------|----------|------|
| preprocessor.py | Python | src/preprocessing/ | 文本预处理器，含 Preprocessor 类和 StopWords 类 |
| stopwords.txt | 数据文件 | data/ | 中英文停用词表 |

##### 2.2.2.2 模块二：医学实体识别模块

**1、简介**

本模块负责从预处理后的文献分词结果中自动识别疾病、药物、症状三类医学实体。采用词典正向最大匹配算法 + jieba 分词双重匹配策略，将识别出的实体按类型标注并去重，输出结构化的实体列表。对应 SRS 需求 FR-03 和 FR-04。

**2、功能列表**

| 功能编号 | 功能名称 | 对应需求ID | 说明 |
|----------|----------|-----------|------|
| F-03 | 医学实体词典管理 | FR-03 | 加载和管理疾病/药物/症状三类词典 |
| F-04 | 实体识别与抽取 | FR-04 | 词典匹配 + jieba 分词联合识别，输出实体列表 |

**（1）类**

**1）EntityDict（实体词典管理器）**

| 属性/方法 | 说明 |
|-----------|------|

## 3 Data Structure 数据结构 / Database Design 数据库设计

### 3.1 概念模型

采用 MySQL 关系型数据库，设计三张表存储文献、实体和关系数据。NetworkX 在内存中构建图结构用于图算法计算。

**MySQL 数据库 E-R 图：**

```
  ┌──────────────────┐       ┌──────────────────┐
  │    article        │       │    entity         │
  │──────────────────│       │──────────────────│
  │ article_id PK    │──┐    │ entity_id PK      │
  │ pmid             │  │    │ name              │
  │ title            │  │    │ entity_type       │  (Disease/Drug/Symptom)
  │ abstract         │  │    │ source            │
  │ authors          │  │    │ article_count     │
  │ journal          │  │    └────────┬─────────┘
  │ pub_year         │  │             │
  │ language         │  │    ┌────────▼─────────┐
  │ source_file      │  │    │    relation       │
  └────────┬─────────┘  │    │──────────────────│
           │            │    │ id PK (自增)      │
           │            ├───►│ source_id FK      │ (article 或 entity)
           │            │    │ target_id FK      │ (entity)
           │            │    │ relation_type     │  MENTIONS/CO_OCCURS/
           │            │    │                   │  TREATS/HAS_SYMPTOM
           │            │    │ frequency         │
           │            │    │ confidence        │
           │            │    └──────────────────┘
           │            │
           ▼            │
  ┌──────────────────┐  │
  │    NetworkX       │◄─┘  ← 系统启动时从 MySQL 加载
  │   内存图          │      ← 图算法计算（最短路径/邻居等）
  └──────────────────┘
```

### 3.2 数据库表设计

#### 3.2.1 表 1：article（文献表）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| article_id | VARCHAR(32) | 是 | 文献唯一标识，主键 |
| pmid | VARCHAR(20) | 否 | PubMed ID |
| title | TEXT | 是 | 文献标题 |
| abstract | TEXT | 否 | 摘要全文 |
| authors | TEXT | 否 | 作者列表 |
| journal | VARCHAR(200) | 否 | 期刊名称 |
| pub_year | INT | 否 | 出版年份 |
| language | VARCHAR(10) | 是 | 语种 en/zh |
| source_file | VARCHAR(100) | 是 | 来源文件名 |

#### 3.2.2 表 2：entity（实体表）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| entity_id | VARCHAR(32) | 是 | 实体唯一标识，主键 |
| name | VARCHAR(200) | 是 | 实体名称 |
| entity_type | VARCHAR(20) | 是 | 实体类型：Disease / Drug / Symptom |
| source | VARCHAR(100) | 否 | 来源词典名称 |
| article_count | INT | 否 | 出现该实体的文献数 |

> **说明：** Disease/Drug/Symptom 三类实体合并为一张 entity 表，通过 `entity_type` 字段区分类型，避免表膨胀。

#### 3.2.3 表 3：relation（关系表）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | INT AUTO_INCREMENT | 是 | 自增主键 |
| source_id | VARCHAR(32) | 是 | 源实体/文献 ID |
| target_id | VARCHAR(32) | 是 | 目标实体 ID |
| relation_type | VARCHAR(20) | 是 | MENTIONS / CO_OCCURS / TREATS / HAS_SYMPTOM |
| frequency | INT | 是 | 共现频次，默认 1 |
| confidence | VARCHAR(10) | 否 | High(≥5) / Low(<5) |

**唯一性约束与索引（SQL DDL）：**

```sql
-- article 表主键与索引
ALTER TABLE article ADD PRIMARY KEY (article_id);
CREATE INDEX idx_article_title ON article (title(100));
CREATE INDEX idx_article_year ON article (pub_year);

-- entity 表主键与索引
ALTER TABLE entity ADD PRIMARY KEY (entity_id);
CREATE INDEX idx_entity_name ON entity (name);
CREATE INDEX idx_entity_type ON entity (entity_type);

-- relation 表索引
CREATE INDEX idx_relation_source ON relation (source_id);
CREATE INDEX idx_relation_target ON relation (target_id);
CREATE INDEX idx_relation_type ON relation (relation_type);
CREATE UNIQUE INDEX idx_relation_unique ON relation (source_id, target_id, relation_type);
```

**数据量预估：**

| 表 | 预估行数 |
|------|----------|
| article | ~700 |
| entity | ~1,000 (Disease 500 + Drug 300 + Symptom 200) |
| relation | ~5,400 |
| 总行数 | ~7,100 |
