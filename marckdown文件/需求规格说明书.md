## 2 General description 总体概述

### 2.1 Software perspective 软件概述

#### 2.1.1 About the Project 项目介绍

医学研究人员在文献调研中面临以下痛点：传统关键词检索无法发现文献间的隐性关联，疾病、药物、症状等实体分散在海量文献中缺乏结构化整合，缺少直观的知识关系可视化工具。

本项目是一个独立的 MVP（Minimum Viable Product）系统，基于自然语言处理（NLP）和知识图谱技术，对预设的医学文献数据集（PubMed 英文文献 500 篇 + 中文医学文献 200 篇）进行结构化解析，自动抽取疾病、药物、症状三类医学实体及其关联关系，构建医学知识图谱，并通过 Streamlit 搭建 Web 界面，提供文献检索和交互式图谱可视化。

项目工期 7 天（2026-07-16 至 2026-07-23），由 3 人团队协作开发，为实训教学项目。

### 2.2 Software function 软件功能

系统功能分为四大模块：

```
┌─────────────────────────────────────────────────────────────┐
│               基于知识图谱的医疗文献管理平台                      │
├───────────────┬───────────────┬───────────────┬─────────────┤
│  文献数据导入  │  医学实体识别  │  知识图谱构建  │  Web 交互界面 │
│  与预处理模块  │     模块      │  与管理模块    │    模块      │
├───────────────┼───────────────┼───────────────┼─────────────┤
│ · 文献加载    │ · 实体词典管理 │ · 图模型设计   │ · 文献检索   │
│ · 文本预处理  │ · 实体识别抽取 │ · 节点批量导入 │ · 图谱可视化 │
│ · 分词流水线  │               │ · 共现关系构建 │ · 文献详情   │
│               │               │ · 查询API封装  │             │
└───────────────┴───────────────┴───────────────┴─────────────┘
```

1. **文献数据导入与预处理模块**：加载预设文献数据集，完成文本清洗、jieba 分词、实体词典加载等预处理。
2. **医学实体识别模块**：基于医学词典 + jieba 分词，自动识别疾病、药物、症状三类实体。
3. **知识图谱构建与管理模块**：将实体及共现关系存入 Neo4j，封装 Cypher 查询 API。
4. **Web 交互界面模块**：基于 Streamlit，提供文献检索、知识图谱可视化、文献详情浏览。

各模块间通过数据文件和 API 接口松耦合协作：预处理流水线输出结构化的实体数据供图谱模块消费；图谱模块封装查询 API 供 Web 前端调用。

### 2.3 User characteristics 用户特征

| 角色 | 特征 | 职责 |
|------|------|------|
| 普通用户（医学研究人员） | 具备医学领域知识，了解基本检索操作，无需编程能力 | 按关键词或实体类型检索文献，浏览知识图谱，查看实体关联网络 |
| 系统管理员（开发团队兼任） | 具备 Python/Neo4j 技术能力 | 文献数据导入，触发实体识别和图谱构建流程，系统维护 |

### 3.2 文献数据导入与预处理模块

#### 3.2.1 文献数据加载

| 属性 | 说明 |
|------|------|
| 需求ID | FR-01 |
| 优先级 | 高 |
| 描述 | 系统支持加载预设的医学文献数据集（PubMed 英文文献 + 中文医学文献），完成数据读取和结构化解析 |
| 输入 | 结构化的文献数据文件（CSV/JSON 格式），包含 title、abstract、authors、journal、pub_year 等字段 |
| 处理 | 读取源文件 → 字段解析 → 缺失值处理 → 数据标准化 → 存储为结构化文献对象列表 |
| 输出 | 结构化的文献对象列表，每篇文献包含完整的元数据字段 |
| 异常处理 | 对格式异常的文献跳过并记录日志，缺失摘要的文献保留但标注为空 |
| 验收标准 | 成功加载 ≥500 篇 PubMed 文献 + ≥200 篇中文文献，加载失败率 ≤ 2% |

#### 3.2.2 文本预处理流水线

| 属性 | 说明 |
|------|------|
| 需求ID | FR-02 |
| 优先级 | 高 |
| 描述 | 对文献标题和摘要进行文本清洗、分词处理，加载医学实体词典，为实体识别做准备 |
| 输入 | 文献的 title 和 abstract 文本字段 |
| 处理 | 去除特殊字符/标点/数字 → 文本标准化（全角转半角、去多余空格） → jieba 分词 → 加载三分类医学实体词典 → 构建分词-词典双向匹配索引 |
| 输出 | 每篇文献的分词结果列表 + 实体词典索引对象 |
| 异常处理 | 空标题/摘要的文献跳过分词步骤，标记为待处理 |
| 验收标准 | 人工抽查 50 篇文献的关键词切分正确率 ≥ 85% |

### 3.3 医学实体识别模块

#### 3.3.1 医学实体词典管理

| 属性 | 说明 |
|------|------|
| 需求ID | FR-03 |
| 优先级 | 高 |
| 描述 | 系统内置医学实体词典，覆盖疾病、药物、症状三类实体，支持词典加载和查询 |
| 数据来源 | ICD-10 疾病分类中文名、DrugBank 药品中文名、百度百科医学词条 |
| 输出 | 实体名 → 实体类型映射表（Disease/Drug/Symptom） |
| 验收标准 | 词典至少覆盖 500+ 疾病名、200+ 药物名、200+ 症状名 |

#### 3.3.2 实体识别与抽取

| 属性 | 说明 |
|------|------|
| 需求ID | FR-04 |
| 优先级 | 高 |
| 描述 | 基于词典正向最大匹配 + jieba 分词，自动从文献标题和摘要中识别疾病、药物、症状三类医学实体 |
| 输入 | 文献的分词结果 + 实体词典索引 |
| 处理 | 词典正向最大匹配 → 实体类型标注 → 同篇文献内去重 → 记录实体在原文中的位置 |
| 输出 | 每篇文献的实体列表 [{entity_name, entity_type, position}] |
| 异常处理 | 无实体匹配的文献输出空实体列表，不中断流水线 |
| 验收标准 | 实体识别 Recall ≥ 70%（以人工标注 50 篇文献为评估基准），单篇文献处理时间 ≤ 0.5 秒 |

### 3.4 知识图谱构建与管理模块

#### 3.4.1 图模型设计

| 属性 | 说明 |
|------|------|
| 需求ID | FR-05 |
| 优先级 | 高 |
| 描述 | 设计 Neo4j 图数据库 Schema，定义节点类型、关系类型及属性字段 |

**节点类型：**

| 节点标签 | 说明 | 关键属性 |
|----------|------|----------|
| Article | 文献节点 | article_id, pmid, title, abstract, authors, journal, pub_year, language |
| Disease | 疾病实体 | entity_id, name, source |
| Drug | 药物实体 | entity_id, name, source |
| Symptom | 症状实体 | entity_id, name, source |

**关系类型：**

| 关系类型 | 源节点 | 目标节点 | 说明 | 属性 |
|----------|--------|----------|------|------|
| MENTIONS | Article | Disease/Drug/Symptom | 文献提及某实体 | frequency |
| CO_OCCURS | Disease/Drug/Symptom | Disease/Drug/Symptom | 两实体共现于同一文献 | frequency, confidence |
| TREATS | Drug | Disease | 药物治疗某疾病 | frequency |
| HAS_SYMPTOM | Disease | Symptom | 疾病有某症状 | frequency |

验收标准：Schema 设计评审通过，节点和关系定义完整清晰。

#### 3.4.2 实体节点导入

| 属性 | 说明 |
|------|------|
| 需求ID | FR-06 |
| 优先级 | 高 |
| 描述 | 将实体识别结果批量导入 Neo4j，创建 Article、Disease、Drug、Symptom 节点 |
| 输入 | 每篇文献的实体识别结果列表 |
| 处理 | 实体跨文献去重合并 → 构建 Cypher MERGE 语句 → 批量执行（每批 500 条） |
| 异常处理 | 单批导入失败时重试 3 次，全部失败则降级为 NetworkX 内存图谱 |
| 验收标准 | 全部文献和实体成功导入，节点数量与去重后实体数一致，批量导入耗时 ≤ 5 分钟（1,700 节点量级） |

#### 3.4.3 共现关系构建

| 属性 | 说明 |
|------|------|
| 需求ID | FR-07 |
| 优先级 | 高 |
| 描述 | 基于文献内实体共现分析，构建实体间的 CO_OCCURS 关系，同时写入 MENTIONS 关系 |
| 输入 | 每篇文献的实体列表 + 已创建的实体节点 |
| 处理 | 同一文献内任意两实体 → 生成共现记录 → 跨文献统计共现频次 → 设置频次阈值 ≥ 2 过滤偶然共现 → 标注置信度（高频 ≥ 5 为 High，其余 Low） → 写入 Neo4j |
| 异常处理 | 频次低于阈值的共现对不创建关系，记录统计信息 |
| 验收标准 | 共现关系总数 ≥ 500 条，图谱可正常查询和展示 |

#### 3.4.4 查询 API 封装

| 属性 | 说明 |
|------|------|
| 需求ID | FR-08 |
| 优先级 | 中 |
| 描述 | 将常用 Cypher 查询封装为 Python 函数，供后端 API 调用 |

预定义查询类型：

| 查询 | 说明 | 响应时间要求 |
|------|------|-------------|
| 按实体名查询关联 | 给定实体名称，返回直接关联的节点和关系 | ≤ 2 秒 |
| 按实体类型 Top-N | 按 Disease/Drug/Symptom 查询 Top-N 实体 | ≤ 2 秒 |
| 两实体最短路径 | 查询两个指定实体之间的最短关联路径 | ≤ 3 秒 |
| 文献关联实体 | 查询某篇文献提及的所有实体 | ≤ 1 秒 |

验收标准：4 类查询正常返回 JSON 结果，响应时间符合要求。

### 3.5 文献检索模块

#### 3.5.1 关键词检索

| 属性 | 说明 |
|------|------|
| 需求ID | FR-09 |
| 优先级 | 高 |
| 描述 | 用户在 Web 界面输入关键词，系统在文献标题和摘要中模糊匹配，返回相关文献列表 |
| 输入 | 用户输入的关键词（支持中英文） |
| 匹配范围 | 文献 title + abstract 字段全文模糊匹配 |
| 输出 | 文献列表（标题、作者、年份、匹配摘要片段高亮），按匹配相关度排序，默认返回 Top 20 |
| 异常处理 | 无匹配结果时提示"未找到相关文献，请尝试其他关键词" |
| 验收标准 | 输入常见医学术语（如"糖尿病""lung cancer"），返回相关文献 ≥ 5 篇，响应 ≤ 3 秒 |

## 4 Requirements Classification 需求分级

| 需求ID | 需求名称 | 需求分级 |
|--------|----------|----------|
| FR-01 | 文献数据加载 | A（必须的） |
| FR-02 | 文本预处理流水线 | A（必须的） |
| FR-03 | 医学实体词典管理 | A（必须的） |
| FR-04 | 实体识别与抽取 | A（必须的） |
| FR-05 | 图模型设计 | A（必须的） |
| FR-06 | 实体节点导入 | A（必须的） |
| FR-07 | 共现关系构建 | A（必须的） |
| FR-08 | 查询 API 封装 | B（重要的） |
| FR-09 | 关键词检索 | A（必须的） |
| FR-10 | 按实体类型筛选 | B（重要的） |
| FR-11 | 交互式图谱展示 | A（必须的） |
| FR-12 | 实体关联路径查看 | C（最好有的） |
| FR-13 | 文献详情浏览 | A（必须的） |

**分级说明：**

- **A（必须的）**：绝对基本的特性；如果不包含，产品就会被取消。共 9 项。
- **B（重要的）**：不是基本的特性，但缺失会影响产品的生存能力和用户体验。共 2 项。
- **C（最好有的）**：期望的特性；省略不会影响产品的生存能力，可在后续迭代中实现。共 1 项。

> **说明：** MVP 版本优先保障 A 级需求 9 项全部交付；B 级需求在时间允许时实现；C 级需求（实体关联路径查看）为附加特性，条件不具备时不做。
