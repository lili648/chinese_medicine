## 1. 项目提出

### 项目名称

基于知识图谱的医疗文献管理平台

### 项目简介

基于知识图谱的医疗文献管理平台是一个智能化的医学文献分析与检索系统（MVP 版本）。系统通过自然语言处理（NLP）和知识图谱技术，对医学文献数据集（PubMed、中文医学文献）进行结构化解析，自动抽取疾病、药物、症状等关键医学实体及其关联关系，构建医学知识图谱。

本阶段（7天）交付的核心能力：

- **文献检索**：按关键词、实体类型（疾病/药物/症状）检索相关文献集合；
- **医学实体识别**：基于词典 + jieba 分词，自动从文献摘要中抽取疾病、药物、症状实体；
- **知识图谱可视化**：以交互式图谱展示疾病—药物—症状—文献之间的关联网络，支持节点点击探索和关联路径查看。

该平台旨在大幅提升医学研究人员的文献调研效率，从传统的关键词匹配检索升级为知识驱动的智能文献管理。

### 项目目标

本项目在 7 天内聚焦以下核心目标（MVP 版本），后续可在此基础上持续迭代：

1. **文献数据预处理**：使用预设的医学文献数据集（PubMed 摘要 + 中文医学文献），完成文本清洗、分词、实体词典加载等预处理工作。
2. **医学实体识别**：基于医学词典 + jieba 分词实现疾病、药物、症状等关键实体的自动识别，确保基本可用。
3. **知识图谱构建与可视化**：将识别出的实体及共现关系存入 Neo4j 图数据库，并通过 Streamlit 搭建 Web 界面，支持文献检索、图谱浏览和实体关联查询。

> 注：基于深度学习的 NER 模型训练、智能推荐、用户管理等为后续迭代目标，不在本次 7 天范围内。

### 系统边界

**系统范围（In Scope）：**

- 预设医学文献数据集（PubMed 500 篇 + 中文医学文献 200 篇）的导入与预处理
- 基于词典 + jieba 的医学实体识别（疾病、药物、症状三类）
- 基于共现分析的实体关系构建
- Neo4j 知识图谱存储与 Cypher 基础查询
- Streamlit Web 界面（文献检索 + 知识图谱可视化）

**不在系统范围（Out of Scope）：**

- 文献自动爬取与实时更新
- 深度学习模型训练（BERT/BioBERT NER、关系抽取模型）
- 语义检索引擎（使用关键词 + 图谱匹配代替）
- 用户登录与权限管理
- 智能推荐系统
- 文献管理（标注、笔记、收藏）

## 4. 风险评估和规避

### 技术风险

**风险1：医学实体词典覆盖率不足**

- 说明：基于词典的实体识别方法依赖词典的完备性，可能存在未收录的疾病、药物名称导致漏识别。
- 解决：
  1. 整合多个公开医学词典（ICD-10、DrugBank 中文版、百度百科医学词条），最大化覆盖率；
  2. 优先保障高频实体的识别，对低频实体允许部分遗漏；
  3. 后续可通过深度学习 NER 模型弥补词典方法不足。

**风险2：共现关系准确性有限**

- 说明：基于共现（同一句/同一摘要中出现）构建的关系存在噪音，部分共现不代表真实因果关系。
- 解决：
  1. 设置共现频率阈值，仅保留多次共现的实体对，过滤偶然共现；
  2. 在界面中标注关系置信度（高频共现 vs 低频共现），供用户判断；
  3. 将此作为首版方案的合理折中，后续可升级为模型抽取。

**风险3：Neo4j + Streamlit 集成复杂度**

- 说明：团队可能缺乏 Neo4j 和 py2neo 的使用经验，环境搭建和 API 对接可能耗时超预期。
- 解决：
  1. 使用 Neo4j Desktop 或 Docker 一键部署，降低配置复杂度；
  2. 提前准备 py2neo 调用模板代码和 Cypher 查询示例；
  3. 如 Neo4j 集成受阻，降级为 NetworkX 内存图谱 + 简单 Web 展示。

### 管理风险

**风险1：7天工期极度紧张**

- 说明：3 人 7 天完成完整的数据—图谱—Web 链路，任何一个环节延期都会影响整体交付。
- 解决：
  1. 每日站会对齐进度，发现问题当天解决；
  2. 严格划定 MVP 边界，不做范围蔓延；
  3. 第 3 天进行中期检查，若进度落后则砍掉非核心功能（如中文文献支持、复杂查询）保底交付；
  4. 预留 1.5 天缓冲（联调 1 天 + 演示准备 1 天）。

**风险2：成员技能不对称**

- 说明：3 人技术栈不完全匹配，可能导致某些模块无人胜任。
- 解决：
  1. Day 1 上午统一进行技术栈快速培训（jieba 分词、Neo4j 基础、Streamlit 入门）；
  2. 按各自强项分工，且每日交叉 review 代码互相 backup；
  3. 准备代码模板和示例工程，降低上手门槛。

### 其它风险

**风险1：数据获取时效风险**

- 说明：PubMed API 在短时间内批量请求可能被限流，中文文献数据需要提前准备。
- 解决：
  1. Day 1 优先启动 PubMed 数据下载，使用带重试机制的批量脚本；
  2. 中文文献使用项目预设数据集，不依赖实时爬取；
  3. 准备一份最小可用数据集（100 篇）作为兜底，确保图谱演示不中断。

**风险2：环境依赖冲突**

- 说明：Neo4j、Python 依赖包可能因环境差异出现安装失败或版本冲突。
- 解决：
  1. 统一使用 requirements.txt 锁定依赖版本；
  2. 优先使用 pip 安装，团队统一 Python 3.9+ 环境；
  3. Neo4j 使用 Docker 部署，规避 Java 版本等环境问题。
